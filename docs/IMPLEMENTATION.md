# Implementation notes: optimization under a transcript-equality invariant

This document records the engineering behind the measured performance and
the methodology used to keep an aggressive optimization campaign honest
about correctness. It is implementation documentation, deliberately kept
out of the research draft (`docs/vole-act.tex`); regression evidence of the
kind described here constrains behavior on sampled inputs — it is not a
proof, and it says nothing by itself about security or privacy.

The optimization campaign and before/after figures below predate the
signer-salted common credential wrapper. That later protocol revision
intentionally changed transcripts, randomness consumption, circuit size, and
performance. Current measurements and a labeled pre-wrapper baseline live in
`docs/BENCHMARKS.md`.

## Methodology

The protocol is deterministic given its RNG stream. Within a fixed protocol
revision, no accepted optimization may alter how randomness is consumed. We
therefore maintained a golden-hash corpus: SHA3-256 digests of 27 protocol
artifacts (keys, requests, responses, tokens, across all three VOLE profiles)
under fixed seeds, generated by
`crates/vole-act/examples/determinism_hash.rs`. Every candidate optimization
was admitted only if:

1. it is arguable as a pure re-expression of identical arithmetic — XOR
   reassociation, bit-matrix transposition, precomputation of public
   values, order-preserving parallelism;
2. the full test suite passes, including malicious-prover soundness
   regressions and kept-in-tree reference oracles for each rewritten
   kernel; and
3. every golden hash is byte-identical, regression-checking the emitted
   transcripts on the sampled corpus.

Changes were benchmarked individually, hypothesis-first; two were rejected
by measurement (below). A second, internal adversarial review pass over the
accumulated diff (distinct from the still-outstanding external audit)
caught two regressions the corpus could not see — a zeroization gap via
container reallocation, and a panic reachable only through a
contract-violating circuit — both fixed and regression-tested.

## What actually dominated

Profiling overturned our priors. Keccak hashing and GF(2^128)
multiplication — the "cryptographic" costs — were minor; the dominant
costs were data reshaping and memory management: (1) assembling
per-coordinate field elements from tau*h bit planes with a branchy per-bit
loop (18% of proving, 37% of verification); (2) roughly 600,000 small heap
allocations per proof in the polynomial-expression layer; (3) per-element
volatile zeroization of large secret buffers — and, embarrassingly, of
3.4 MB of *public* quadratic-system coefficients per proof. The effective
fixes, in decreasing order of impact:

- **Word-level bit transposes.** Plane assembly and the consistency-hash
  column gathers are 128x64 / 128x128 bit-matrix transposes; the standard
  word transpose replaced per-bit loops, incidentally removing a
  secret-dependent branch.
- **Inline expression storage.** Degree-<=16 coefficient vectors live in
  fixed inline arrays; constraint storage is pre-sized from the counting
  pass so growth reallocation can never copy secrets into unwiped freed
  memory.
- **A shared arena for public coefficients.** Quadratic-system coefficient
  tables are public-key material; storing them once (reference-counted
  views instead of per-term vectors) removed both the allocation traffic
  and their spurious per-proof zeroization.
- **Parallel and cheaper tree expansion.** The tau GGM trees commit and
  reconstruct in parallel with order-preserving collection; the u/plane
  sums use a streaming binary-counter accumulation (pairwise subtree sums,
  ~2 vector XORs per leaf instead of 1 + h/2).
- **Hardware and SWAR kernels.** Keccak uses the ARMv8.2-SHA3 extension
  where available; GF(16) matrix kernels run eight lanes per word with
  mask-based selection (no lookup tables, preserving the constant-time
  contract); Delta-powers are precomputed; the Keccak rho/pi permutation is
  fused into chi's indexing.

In the historical pre-wrapper revision, the cumulative result on an Apple M4
Pro (balanced profile) was: issuer verify-and-sign 21.4 -> 6.5 ms,
direct-spend end-to-end 66.0 -> 26.4 ms, compact client proving
59.4 -> 20.2 ms. The compact profile's proving penalty over low-latency
collapsed from 1.4x to ~6-7%, making the smallest proofs a strong default
candidate (the shipped default remains balanced).

## Negative results

Recorded so they are not re-attempted from intuition:

1. **Whole-struct volatile zeroization is slower than per-field.**
   Replacing per-field volatile stores with a whole-struct volatile write
   (`zeroize_flat_type`) measured *slower*: LLVM lowers large volatile
   aggregate stores poorly.
2. **A specific AES-CTR leaf PRG lost to hardware SHAKE.** AES-128-CTR
   keyed per leaf (RustCrypto `aes` 0.8 / `ctr` 0.9, release build with
   LTO, Apple M4 Pro hardware AES) producing 4.2 KB per 16-byte seed
   measured 2.4x slower than SHAKE256 once the ARMv8.2-SHA3 extension is
   enabled, for this workload shape. The construction tested also does not
   by itself provide the One-Tree salt-and-position binding, which any AES
   instantiation must re-engineer via key derivation, tweaks, or extra
   calls — FAEST v2's AES-based leaf commitments show this is achievable,
   at design-and-analysis cost. Since the properly hardened variant starts
   from a 2.4x deficit on this hardware, the hash-based PRG stays; the
   hardening itself is a requirement, not a trade-off.

## Tooling

- `examples/determinism_hash.rs` — the transcript-compatibility regression
  harness; re-run and diff against a captured baseline around any
  optimization.
- `examples/profile_matrix.rs` — every operation and wire size across all
  three profiles (medians of 15).
- `examples/profile_hotpath.rs` — infinite-loop harnesses for sampling
  profilers (`prove`, `verify`, `pkdecode` modes; optional profile arg).

Benchmarking on macOS: run under `caffeinate -disu`, and check that load is
low and WindowServer is quiet first — idle-display wallpaper animation has
corrupted unattended benchmark runs on this hardware.
